Close the single surviving Phase 4 UAT gap (test 2, severity major, 9/10 passed): after a provider switch in Settings → AI Features, the Model select ends up EMPTY — the trigger shows the "Select a model" placeholder and Save is blocked by zod validation until the operator manually re-picks a model. This contradicts 04-04's documented decision D-01 ("Provider changes reset the model selection to the new provider's first catalog entry").

Root cause is fully diagnosed (`.planning/debug/model-select-clears-on-provider.md`, status: diagnosed): an upstream @radix-ui/react-select@2.3.1 bug (radix-ui/primitives #3135/#3381/#3693). The Select sits inside a `<form>`, so Radix renders SelectBubbleInput — a hidden UNCONTROLLED native `<select>` whose `<option>`s update one commit AFTER the SelectItem children swap. handleProviderChange's batched setValue calls set the value to the new catalog[0] while the native select still holds the OLD provider's options; the bubble-input sync effect assigns the new value against non-matching options → per HTML spec `select.value` becomes "" → it dispatches a change event → `onValueChange("")` → `field.onChange("")` clobbers the setValue.

Purpose: AIDA-13's acceptance ("selectable in settings") is only fully true when a provider switch leaves the form saveable; this is the last gap between Phase 4 and full UAT pass.
Output: a one-line component fix (`key={provider}` on the Model `<Select>`), the T2/T10 explicit-pick workarounds in the E2E spec reverted to assert auto-reset directly, and the full phase-4 E2E spec green 11/11.

    Why this works (from the diagnosed debug session): keying the Model Select by `provider`
    remounts the entire Radix Select subtree in the SAME React commit where
    `handleProviderChange` has already set `field.value` to the new catalog's first entry.
    The initial-mount path is proven safe — `usePrevious` returns the current value on first
    render so the SelectBubbleInput sync effect never dispatches, and the hidden native
    select remounts fresh once the NEW options register. The stale-options race
    (old provider's `<option>`s + new value → `select.value === ""` → `onValueChange("")`
    clobbering the setValue) becomes structurally impossible.

    Constraints — deviations here reintroduce bugs:
    - Do NOT key the Provider `<Select>` (line 166): its items are static so it is not
      affected by the race, and keying it would remount/drop focus on every change.
    - Do NOT modify `handleProviderChange` — its three setValue calls are correct; the
      empty-string clobber arrives from Radix, not from react-hook-form batching
      (explicitly eliminated in the debug session).
    - Do NOT guard `onValueChange` against `""` or patch/pin @radix-ui/react-select —
      the one-line key remount is the diagnosed minimal fix.
