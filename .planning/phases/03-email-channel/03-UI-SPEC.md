# Phase 3 — UI Design Contract: Email Channel

> Visual and interaction contract for the new Settings "Email" tab and the per-message outbound-delivery-failure affordance. Generated by gsd-ui-researcher, verified by gsd-ui-checker. Binding source of truth: `.planning/DESIGN-SYSTEM.md` (indigo-violet, established Phase 1) + `.planning/phases/03-email-channel/03-CONTEXT.md` (D-01..D-26) + `.planning/phases/03-email-channel/03-RESEARCH.md`.
>
> **Scope note:** this phase's UI surface is deliberately narrow (per CONTEXT D-21/D-24/D-25) and additive to an already-established Settings pattern. Every token, component, and layout convention below already exists in the shipped codebase (Phase 1/2) — this contract's job is to specify **which existing pattern applies where**, not to invent new visual language. No new CSS tokens, no new shadcn components, and only two genuinely new (but fully precedent-based) interaction patterns: the "Test connection" button and the "Failed to send — Retry" chip.
>
> **Explicitly out of scope for this contract:** the outbound HTML email wrapper (RESEARCH.md D-22, `wrapEmailSafeHtml()`). That markup is rendered inside third-party mail clients (Gmail, Outlook, Apple Mail), which cannot load Tailwind or CSS custom properties — it correctly uses **inline literal styles**, not design tokens. Do not apply this file's token rules or the DESIGN-SYSTEM.md checklist to that wrapper; RESEARCH.md already flags this distinction.

## Screen Contracts

### 1. Settings → Email tab — new `/settings/email`

**Nav:** add one entry to the existing `navItems` array in `src/app/(app)/settings/settings-nav.tsx`:
```ts
{ href: "/settings/email", label: "Email" }
```
No other change to `SettingsNav` — the existing active/inactive class logic (`bg-primary/10 text-primary` / `text-muted-foreground hover:bg-muted`) and `startsWith` matching already handle a new sub-route correctly.

**Page shell** — identical to every existing Settings page:
```tsx
<div className="space-y-6">
  <h1 className="text-[18px] font-semibold">Email</h1>
  {/* EmailChannelToggle, then the two sections */}
</div>
```

**Authorization shape** (matches existing SLA/Tags/Custom-Fields precedent exactly): the page itself renders for any authenticated org member (no page-level route guard — consistent with `sla/page.tsx`, `tags/page.tsx`, `custom-fields/page.tsx`, none of which call `requireOrgAdmin()` directly). Every **mutating** Server Action (`saveEmailSettings`, `testImapConnection`, `testSmtpConnection`) calls `requireOrgAdmin()` first, per D-24 and `src/lib/authz.ts`'s documented contract ("every mutating Settings Server Action must call this first"). This reading is flagged explicitly in Assumptions for planner confirmation since D-24's phrasing ("admin-gated Email tab") could be misread as a page-level guard.

**`EmailChannelToggle`** (top of page, above both sections) — verbatim `AiToggle` shape:
```tsx
<div className="flex flex-col gap-2">
  <div className="flex items-center gap-3">
    <Switch checked={enabled} onCheckedChange={handleChange} aria-label="Enable email channel" />
    <span className="text-[14px]">Enable email channel</span>
  </div>
  <p className="text-[12px] text-muted-foreground">
    Poll a mailbox for inbound email and send agent replies via SMTP. Off by default until configured.
  </p>
</div>
```

**IMAP section** (`space-y-4 rounded-lg border border-border/70 p-4`):
- Eyebrow: `<p className="text-[12px] font-medium uppercase tracking-wide text-muted-foreground">IMAP (Inbound)</p>`
- `EmailHealthLine` (D-25) — rendered directly below the eyebrow, above the fields (first thing an admin checking channel health sees):
  - Healthy, ever-polled: `text-[12px] text-muted-foreground` — `"Last checked {formatRelativeTime(lastPollAt)}"` (reuses `src/lib/format-relative-time.ts` `formatRelativeTime()` verbatim, same helper `ThreadMessage` already uses).
  - Healthy, never polled (channel just enabled): `"Not checked yet — the inbound poll runs every minute once enabled."`
  - Failing (`lastPollError` present) — reuses the exact destructive-banner shape already proposed in `02-UI-SPEC.md` §1:
    ```tsx
    <div className="flex items-center gap-2 rounded-lg border border-destructive/20 bg-destructive/10 p-3 text-[13px] text-destructive">
      <CircleAlert className="size-4" />
      Last poll failed: {lastPollError}
    </div>
    ```
- Fields (`space-y-4`, each an existing `FormField`+`FormLabel` (`text-[13px] font-normal text-muted-foreground`) + `Input`): Host, Port (`type="number"`, placeholder `993`), "Use SSL/TLS" (`Checkbox`, matching `custom-field-input.tsx`'s CHECKBOX pattern), Username, Password (`Input type="password"`, no show/hide toggle — v1 simplification, flagged in Assumptions).
- `TestConnectionButton` (below the fields): idle label `"Test IMAP connection"`.

**SMTP section** — identical shape/spacing to the IMAP section, no `EmailHealthLine` (health tracking is inbound-poll-only per D-25):
- Eyebrow: `"SMTP (Outbound)"`.
- Fields: Host, Port (`type="number"`, placeholder `587`), "Use SSL/TLS" (`Checkbox`), Username, Password (`Input type="password"`).
- `TestConnectionButton`: idle label `"Test SMTP connection"`.

**From-address field** — top-level, not nested in either section (applies to outbound headers per D-23):
- `FormField` + `Input type="email"`, label `"From address"`, caption (`text-[12px] text-muted-foreground`): `"The address customers see and reply to. Must match your SMTP account or an address it's authorized to send as."`

**`TestConnectionButton`** (one component, two instances — IMAP and SMTP; new pattern, fully composed from existing primitives):

| State | Rendering |
|-------|-----------|
| Idle | `<Button type="button" variant="outline" size="sm">Test IMAP connection</Button>` (or "Test SMTP connection") |
| Testing | Same button, `disabled`, `<Loader2 className="mr-2 size-3.5 animate-spin" />` + label `"Testing…"` — identical spinner convention to every async-submit button already in this codebase |
| Success | Button returns to idle; inline row appears below it: `<div className="mt-2 flex items-center gap-1.5 text-[12px] text-success"><CheckCircle2 className="size-3.5" />Connected successfully</div>` |
| Failure | Button returns to idle; inline row: `<div className="mt-2 flex items-center gap-1.5 text-[12px] text-destructive"><XCircle className="size-3.5" />Connection failed: {reason}</div>` — `{reason}` is the server-returned error message, truncated; never echoes the submitted password back |

Per RESEARCH.md Pitfall 5, the underlying Server Action applies an explicit 8–10s connection timeout — this is a backend concern, but its UI consequence is binding: the button must never spin indefinitely; a slow/misconfigured host resolves to the Failure state within ~10s, not a stuck spinner.

**Save action** — one form-level submit button at the bottom (mirrors `SlaForm`'s single-submit-button convention, not per-section saves):
```tsx
<Button type="submit" disabled={form.formState.isSubmitting}>
  {form.formState.isSubmitting && <Loader2 className="mr-2 size-4 animate-spin" />}
  Save email settings
</Button>
```

**States:**
- Save success: `toast.success("Email settings saved.")` (matches `SlaForm`'s `toast.success("SLA targets saved.")` exactly).
- Save failure: `toast.error("Failed to save email settings. Please try again.")` (matches the exact `"Failed to {action}. Please try again."` phrasing used by `ai-toggle.tsx`, `tag-manager.tsx`, `sla-form.tsx` — established project-wide convention, not a new phrase pattern).
- Channel disabled (`enabled = false`): the two config sections remain visible and editable (an admin must be able to configure before flipping the switch on) — no dimming/disabling of the form fields; only the health line and the Test Connection buttons are meaningfully affected by the toggle (a disabled channel simply won't have a `lastPollAt` to show), per D-26 "fully optional/toggleable, everything else keeps working."

### 2. Thread message — "Failed to send — Retry" (D-21)

Applies **only** to outbound **public-reply** messages where `deliveryStatus === "FAILED"`. Never applies to internal notes (D-21 is about customer-facing SMTP delivery) or inbound messages.

Extend `ThreadMessage`'s existing outbound-public variant (`src/components/tickets/thread-message.tsx`) with an optional `deliveryStatus?: MessageDeliveryStatus | null` prop. When it equals `"FAILED"`, render one additional row after the existing attachments row (or directly after the message body if there are no attachments):

```tsx
<div className="mt-1.5 flex items-center gap-2">
  <Badge className="h-5 px-2 py-0.5 text-[12px] bg-destructive/10 text-destructive border border-destructive/20">
    <CircleAlert className="size-3" />
    Failed to send
  </Badge>
  <Button variant="link" size="sm" className="h-auto p-0 text-[12px]" onClick={onRetry}>
    Retry
  </Button>
</div>
```

The badge reuses `SlaDueChip`'s exact "Overdue" class string (`h-5 px-2 py-0.5 text-[12px] bg-destructive/10 text-destructive border border-destructive/20` + `CircleAlert` at `size-3`) — no new chip visual is invented. The `Retry` action uses `Button variant="link"`, an existing variant (`button.tsx`) specifically suited to an inline text-style action.

**States:**
- `QUEUED` / `SENT`: render with **zero additional chrome** — current behavior is unchanged. D-21 requires visibility specifically for the FAILED state, not a full send-status indicator system; adding a persistent "Sent ✓" badge to every outbound message would be new visual noise this phase does not require.
- Retry clicked: optimistically replace the chip+button row with a transient `<span className="text-[12px] text-muted-foreground">Retrying…</span>` (no icon — matches this codebase's minimal, toast-driven feedback convention rather than inventing a new spinner-in-thread pattern). The Server Action re-enqueues the send job and calls `revalidatePath` (the existing mutation convention used by every ticket action in `[id]/actions.ts`); the row then reflects the next real `deliveryStatus` on refresh — either it disappears (`SENT`) or reappears (`FAILED` again, permitting another retry).

## Assumptions (review these)

Decisions made at the researcher's discretion where CONTEXT.md/RESEARCH.md did not fully pin down the exact visual shape — flagged for human review rather than blocking:

1. **"Admin-gated Email tab" (D-24) is read as "admin-gated actions", not a page-level route guard** — matching the exact existing precedent of `sla/page.tsx`/`tags/page.tsx`/`custom-fields/page.tsx`, none of which block page rendering by role; only their Server Actions call `requireOrgAdmin()`. Flag for planner/executor to confirm this reading is intended before implementation, since it's the one place this contract infers behavior rather than citing an exact existing line of code.
2. **`TestConnectionButton`'s exact shape** (outline button + below-button inline success/failure row) is a new pattern with no byte-for-byte precedent in the codebase — it composes only from already-shipped primitives and the already-established `Loader2` spinner convention, but the specific "result row appears below the button" layout is this researcher's proposal. Reconsider if the checker or a human reviewer prefers the result inline beside the button instead of below it.
3. **`EmailHealthLine` placement** (directly under the IMAP section eyebrow, above the fields) is this researcher's choice — an alternative would be placing it at the very top of the page, above the channel toggle, for maximum visibility. Kept under IMAP specifically because `lastPollAt`/`lastPollError` are inbound-poll-specific (D-25), not channel-wide.
4. **Password fields use plain `Input type="password"` with no show/hide toggle** — v1 simplification, avoids introducing a new component for a single use case. Flag as a candidate follow-up if UAT surfaces friction (e.g., an admin needing to verify a pasted password before Test Connection).
5. **`DeliveryFailedChip`'s Retry action uses `Button variant="link"` inline next to the badge**, rather than a small icon-only retry button — chosen for label clarity ("Retry" as a word reads more clearly than a bare `RotateCw` icon in a dense thread view) and because no icon-only retry precedent exists elsewhere in this codebase to reuse instead.
6. **No visual treatment for `QUEUED`/`SENT` delivery states** — deliberately minimal, since D-21's binding requirement is specifically about FAILED visibility ("failures are surfaced, not silent" — Success Criterion 3), not a full delivery-status-for-every-message system. Reconsider only if UAT feedback indicates agents want to see in-flight `QUEUED` state (e.g., for a slow SMTP server).
