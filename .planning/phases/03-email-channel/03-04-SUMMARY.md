- `createTicket()` extended (never a second entrypoint, D-10) to carry email identity fields (`emailMessageId`/`emailInReplyTo`/`emailReferences`) and an optional pre-sanitized `bodyHtml`; `CreateTicketResult` now returns `messageId` so ingest can attach files to the initial message.

- `pollInbox()` — enumerates orgs with the channel enabled (bare `prisma`), fetches UNSEEN per org (`client.search({ seen: false }, { uid: true })`), derives the id once per message, checks/increments a poison-guard record (`EmailIngestFailure`, threshold 5), ingests, marks `\Seen` only after success, and persists `lastPollAt`/`lastPollError` health via `updateEmailHealth` (D-25) — failures are never silent.
- `worker/index.ts` now registers both email jobs: `email-inbound-poll` (singleton policy, `* * * * *`) and `email-outbound-send` (`retryLimit: 2`, matching `boss-client.ts`'s queue options) — this is the first plan to touch this file's queue graph with `@/`-transitive imports (via `createTicket`), and the exact Dockerfile esbuild bundle command was re-run as a hard stop and succeeded.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] `thread-match.ts`'s `ThreadMatchDb` type didn't structurally accept a real `scopedDb()` client**
- **Found during:** Task 1 (`ingestMessage()` calling `findTicketIdByEmailMessageIds(db, ...)` / `findTicketByNumber(db, ...)` with the actual scoped Prisma client for the first time — 03-03's own tests only exercised these helpers with hand-typed fakes, never a real client)
- **Issue:** `ThreadMatchDb`'s `findFirst` members were declared as function-typed *properties* (`findFirst: (args: unknown) => ...`), which TypeScript checks under strict, non-bivariant parameter contravariance. The real `scopedDb()` client's `message.findFirst`/`ticket.findFirst` require a specific `*FindFirstArgs` object, not `unknown`, so passing the real client failed `tsc --noEmit` with two `TS2345` errors.
- **Fix:** Changed both members to method-shorthand syntax (`findFirst(args: unknown): ...`) inside the same `ThreadMatchDb` type literal — this switches TypeScript to its (correct, standard) bivariant method-parameter checking, which accepts the real client's narrower parameter type. No behavior change; `ThreadMatchDb` is an internal, unexported type only used to narrow the `db` parameter of `findTicketIdByEmailMessageIds`/`findTicketByNumber`.
- **Files modified:** `src/lib/channels/email/thread-match.ts`
- **Verification:** `pnpm exec tsc --noEmit` clean; `tests/unit/email-thread-match.test.ts` (03-03's existing suite) still 6/6 green; `tests/integration/email-ingest.test.ts`'s subject-token/header-match scenarios (which exercise these functions against a real client) pass.
- **Committed in:** `fd0ba14` (Task 1 commit)
