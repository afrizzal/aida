## Typography

Inherited scale (DESIGN-SYSTEM.md §3) — Phase 2 introduces **no new sizes**, mapped to template roles:

| Role | Size | Weight | Line Height | Phase 2 usage |
|------|------|--------|-------------|----------------|
| Label | 12px | 400 (regular) | 1.4 | Timestamps, captions, tag chip text, avatar fallback, chip labels |
| Micro-label | 13px | 500 (medium) | 1.4 | Pill tabs (filter chips, settings sub-nav), ticket-list contact-row name, key/value `dl` lists (contact detail header), rate-limit banner text |
| Body | 14px | 400 (regular) | 1.5 | Ticket list subject/contact, thread message text, composer textarea, form fields, settings body copy |
| Body (emphasis) | 14px | 500 (medium) | 1.5 | Unread/new ticket subject, contact name in list rows, active filter chip label |
| Brand mark | 15px | 600 (semibold) | 1.2 | Public page brand wordmark ("AIDA" in `PublicPageShell` header) |
| Heading | 18px | 600 (semibold) | 1.2 | Page titles ("Inbox", "Contacts", "Settings"), reading-pane ticket subject, contact detail name, public page Card title |

Font weights used: **400, 500, 600** — three weights, not the generic two-weight guideline. This is an explicit, documented deviation: DESIGN-SYSTEM.md §3 already establishes all three system-wide (Phase 1, e.g. sidebar user name at 500) and this phase introduces no new weight. Treat 600 as reserved for headings/page titles only; 500 for emphasis/active states; 400 for everything else.

No new font sizes are introduced this phase (deliberate — reusing the existing 12/13/14/15/18px inherited scale keeps the app visually consistent rather than adding a "hero" size for the public pages).

## New Design Tokens to Add

Two semantic colors are required this phase that DESIGN-SYSTEM.md references only as inline suggestions (§4.7) but that do not yet exist as CSS custom properties. Per the token-only rule, these must be added to `globals.css` **before** any component uses them — flagging both here rather than inventing raw oklch values inline in components.

| Token | Role | Reserved for |
|-------|------|---------------|
| `--warning` | Amber semantic color (text + tint source) | "Pending" status chip, "At risk" SLA chip, "High" priority chip, Internal Note composer/message surface + lock icon |
| `--success` | Emerald semantic color (text + tint source) | "Resolved" status chip |

Follows the exact existing `--destructive` convention in this codebase (single token, no separate `-foreground` token, used directly as `text-{token}` and as a tint source via `bg-{token}/10`, `border-{token}/20` — confirmed via `badge.tsx`'s `destructive` variant, which has no companion `--destructive-foreground`, and via `button.tsx`'s `destructive` variant, which is likewise tint-only with no solid fill). No `--warning-foreground` or `--success-foreground` token is declared or used anywhere in this spec — every warning/success surface (chips, composer, thread message) uses the tint pattern (`bg-{token}/N text-{token} border border-{token}/N`), never a solid fill requiring a foreground companion.

**Proposed values** (starting point — verify contrast visually per DESIGN-SYSTEM §7 dark-mode check before merging):

```css
/* :root — add alongside --destructive */
--warning: oklch(0.6 0.16 70);   /* amber-700-ish: legible as text on bg-warning/10 */
--success: oklch(0.58 0.14 155); /* emerald-700-ish: legible as text on bg-success/10 */

/* .dark — add alongside --destructive */
--warning: oklch(0.75 0.16 75);  /* brighter for dark-bg legibility, mirrors --destructive's +0.07 lightness lift */
--success: oklch(0.72 0.15 155);
```

Also register in the `@theme inline` block: `--color-warning: var(--warning);` and `--color-success: var(--success);` (same pattern as `--color-destructive`).

**`Badge` component variant additions** (`src/components/ui/badge.tsx`) — mirror the `destructive` variant exactly:

```ts
warning: "bg-warning/10 text-warning focus-visible:ring-warning/20 dark:bg-warning/20 dark:focus-visible:ring-warning/40 [a]:hover:bg-warning/20",
success: "bg-success/10 text-success focus-visible:ring-success/20 dark:bg-success/20 dark:focus-visible:ring-success/40 [a]:hover:bg-success/20",
```

`Badge` size (`h-5 px-2 py-0.5 text-[12px]`), no icon.

| Status | Classes | Label |
|--------|---------|-------|
| `new` | `bg-primary text-primary-foreground` (solid — most attention-grabbing, unaddressed) | "New" |

### PriorityChip (4 levels — D-05)
Shown in the reading-pane header always (all 4 levels, agent-editable via `DropdownMenu`). Shown in the ticket-list row **only for High/Urgent** (deliberate decluttering — Low/Normal are the majority case and would add visual noise to every row; see Assumptions).

| Priority | Classes | Label |
|----------|---------|-------|
| `low` | `border border-border text-muted-foreground` (outline, no dot) | "Low" |
| `normal` | `border border-border text-foreground` (outline, no dot) | "Normal" |
| `high` | `bg-warning/10 text-warning border border-warning/20` | "High" |
| `urgent` | `bg-destructive/10 text-destructive border border-destructive/20` | "Urgent" |

### SlaDueChip (3 states — D-15, exact copy per CONTEXT.md)
Icon `size-3` inline before label (matches `badge.tsx`'s `[&>svg]:size-3!` rule).

| State | Classes | Icon | Label |
|-------|---------|------|-------|
| On track | `bg-muted text-muted-foreground border border-border` | `Clock` | `Due in {duration}` (e.g. "Due in 2h", "Due in 45m") |
| At risk | `bg-warning/10 text-warning border border-warning/20` | `AlertTriangle` | `At risk` |
| Overdue | `bg-destructive/10 text-destructive border border-destructive/20` | `CircleAlert` | `Overdue` |

Duration formatting: relative, coarse-grained (`45m`, `2h`, `1d`) — no seconds. Full timestamp on hover via native `title` attribute.

### TagChip (D-16)
`Badge variant="secondary"` (`bg-secondary text-secondary-foreground`), rounded-full, with an inline `x` remove icon (`size-3`, shown only in edit contexts — reading pane, not list rows); icon-only, requires `aria-label="Remove tag {tag}"`. "+N" overflow chip (`variant="outline"`) when a list row has more than 2 tags.

### Assignee Avatar
`Avatar size="sm"` (24px) in list rows, `size="default"` (32px) in the reading-pane header. Fallback styling matches the existing sidebar pattern exactly: `bg-primary/10 text-primary text-[12px] font-medium` with initials. **Unassigned** state: no avatar — a dashed-outline circular placeholder (`size-6 rounded-full border border-dashed border-border`) containing a `UserPlus` icon (`size-3 text-muted-foreground`), `title="Unassigned"`.

## Screen Contracts

### 1. Shared Inbox (2-pane) — `/tickets`, `/tickets/[id]`

Two-column layout inside the existing `(app)` content area (D-01, no third column). List column fixed width `w-[360px]`, reading pane `flex-1`. Both columns scroll independently (`overflow-y-auto`), each with its own sticky header at `h-14` for vertical alignment. Focal points: the reading-pane subject heading and the composer's primary CTA are the two anchors that draw the eye first.

**Filter row** (sticky top of list column, `h-14 px-4`, `border-b border-border`, `bg-background/80 backdrop-blur-sm` — same sticky/blur treatment as `TopBar` per DESIGN-SYSTEM §4.2):
- Fixed tabs (D-02): `Unassigned` | `Mine` | `All` — pill buttons (`h-8 px-3 rounded-full text-[13px] font-medium`), active = `bg-primary/10 text-primary`, inactive = `text-muted-foreground hover:bg-muted`.
- Status filter: compact `DropdownMenu` button (`Filter` icon + "Status" label + count badge if active) — multi-select checkboxes for the 5 statuses.
- Search input: `Input` with `Search` icon prefix, placeholder `"Search tickets…"`, right-aligned in the filter row or full-width below on narrow viewports (full-text search, AIDA-02).
- Tag filter: `Popover` + `Command` combobox, same pattern as `TagAutocomplete` (D-16 filterability).

**Ticket list row** (`TicketListRow`, min-height 80px, `px-4 py-3`, `border-b border-border/70`):
- Line 1: contact name (`text-[13px] font-medium text-foreground truncate max-w-[70%]`) left; relative timestamp (`text-[12px] text-muted-foreground`) right.
- Line 2: subject (`text-[14px] truncate`; `font-medium` if status = `new`/unread, else `font-normal`).
- Line 3 (chip row, `flex items-center gap-1.5`): `StatusChip`, `PriorityChip` (High/Urgent only), `SlaDueChip`, up to 2 `TagChip`s + overflow, then `ml-auto` Assignee Avatar.
- Selected state: `bg-accent border-l-2 border-l-primary` (matches `DropdownMenu`/`Button` `aria-expanded` active-state token usage elsewhere).
- Hover: `hover:bg-muted/50`.

**Reading pane:**
- `TicketMetaHeader` (`h-14 px-6 sticky top-0 border-b border-border bg-background/80 backdrop-blur-sm`): `#{number}` + subject (`text-[18px] font-semibold truncate`), then right-aligned cluster: Status `DropdownMenu` trigger (shows `StatusChip`), Priority `DropdownMenu` trigger (shows `PriorityChip`), Assignee `DropdownMenu` trigger (shows Avatar + name), `SlaDueChip`.
- Secondary meta row (`px-6 py-2 border-b border-border/70 flex flex-wrap items-center gap-2`): tag chips + `TagAutocomplete` "+ Add tag" ghost button, followed by any custom-field values rendered inline as `Label: value` pairs (compact, `text-[12px]`), each editable via click-to-open `DropdownMenu`/`Popover` per its `CustomFieldInput` type.
- Thread body (`flex-1 overflow-y-auto px-6 py-4 space-y-4`): see §2.
- `Composer` pinned at bottom (`border-t border-border`): see §2.

**States:**
- Loading: 6× `Skeleton` rows in the list column (matching the 80px row height, 3 skeleton bars each); reading pane shows a single centered `Skeleton` block set.
- Empty (workspace has zero tickets ever): reuse `EmptyState` pattern exactly (DESIGN-SYSTEM §4.3) — `Inbox` icon, heading `"Your inbox is empty"`, body `"New tickets will appear here as customers reach out through your web form. Share your intake link to start receiving requests."` Rendered in the reading-pane area (list column shows nothing above the fold, or is hidden/collapsed to full-width empty state — Claude's discretion, list column may collapse to 0 width when there is truly nothing to show).
- Empty (filtered view has zero matches, e.g. "Unassigned" with none): smaller inline empty state within the list column only (not full `EmptyState` halo pattern — too heavy for a filtered sub-state), icon + text centered, `py-12`: `CheckCircle2` icon (`size-5 text-muted-foreground`), text `"Nothing here — no tickets match this view."`
- No ticket selected (list has tickets, none open): reading pane shows `EmptyState` pattern, `MessageSquare` icon, heading `"Select a ticket"`, body `"Choose a ticket from the list to view the conversation."`
- Error (list fails to load): inline banner at top of list column, `bg-destructive/10 border border-destructive/20 rounded-lg p-3 text-[13px] text-destructive`, text `"Couldn't load tickets."` + inline `Button variant="link"` `"Retry"`.
- Long lists: cursor-based pagination, `"Load more"` `Button variant="outline"` centered at the bottom of the list column, default page size 50 (Claude's discretion per CONTEXT — see Assumptions; no virtualization in v1).

### 2. Ticket Detail / Conversation Thread

**ThreadMessage** — chronological, three visual variants sharing one row shell (`flex gap-3`):

| Variant | Container | Label |
|---------|-----------|-------|
| Inbound (from contact) | `bg-card border border-border rounded-lg p-3` | Contact name + relative time, no explicit "public" label needed |
| Outbound (agent public reply) | `bg-primary/5 border border-primary/15 rounded-lg p-3` | Agent name + relative time |
| Internal note | `bg-warning/10 border border-warning/30 border-l-2 border-l-warning rounded-lg p-3` | `Lock` icon (`size-3.5 text-warning`) + `"Internal Note"` (`text-[12px] font-medium uppercase tracking-wide text-warning`) + agent name + relative time |

Message body: rendered sanitized HTML from `renderMarkdown()` (RESEARCH.md Topic 3) inside a `prose`-like wrapper constrained to `text-[14px] leading-relaxed text-foreground` — no Tailwind Typography plugin needed for v1; base element styling (`p`, `strong`, `code`, `ul`/`ol`, `a`) inherits from `globals.css` `@layer base` with minimal additions (`a` → `text-primary underline-offset-2 hover:underline`; `code` → `bg-muted rounded px-1 py-0.5 text-[13px]`).

Attachments: `AttachmentChip` row below message body (`flex flex-wrap gap-1.5 mt-2`) — `bg-muted border border-border rounded-md px-2 py-1 text-[12px] flex items-center gap-1.5`, file-type icon (`Paperclip` default, or a type-specific Lucide icon), truncated filename, size in parens (e.g. `"invoice.pdf (240 KB)"`), click opens the authenticated `/api/attachments/[id]` route in a new tab.

**Composer** (D-10, bottom-pinned, `border-t border-border p-4`):
- `ComposerToggle` (bespoke, not shadcn `Tabs`): two `Button`s side-by-side in a bordered `rounded-lg` container (`inline-flex border border-border rounded-lg overflow-hidden`):
  - "Public Reply" — active state `bg-primary/10 text-primary`, inactive `text-muted-foreground hover:bg-muted`.
  - "Internal Note" — active state `bg-warning/10 text-warning`, inactive `text-muted-foreground hover:bg-muted`. **Never** uses primary/indigo styling, even hover, to keep the two modes unmistakably distinct (D-10 requirement).
- Below the toggle: `Textarea` (Markdown source, `min-h-[96px]`), container background shifts with mode: Public = `bg-background`, Internal Note = `bg-warning/5 border border-warning/30 rounded-lg` wrapping the textarea.
- Toolbar row (`flex items-center justify-between mt-2`): left — `Paperclip` icon `Button variant="ghost" size="icon-sm"` (attach file, `aria-label="Attach file"`, triggers native file input, enforces 10MB/allowlist client-side per D-23 with server-side re-check); right — primary action button:
  - Public mode: `Button` default (primary indigo) — label `"Send Reply"`.
  - Internal Note mode: `Button` with warning-tint override, matching the codebase's existing tint-only destructive/warning convention (`badge.tsx`/`button.tsx` — no solid-fill destructive or warning button variant exists in this codebase, and no `--warning-foreground` companion token is declared): `bg-warning/10 text-warning border border-warning/30 hover:bg-warning/20`, still `size="default"` shape, `Lock` icon (`size-3.5`) + label `"Save Internal Note"`. (Deliberately never the indigo primary button in this mode — reinforces D-10's visual-distinction requirement all the way to the CTA.)
- Attached-but-unsent files: shown as removable `AttachmentChip`s above the toolbar row.

**States:**
- Sending: primary/action button shows a spinner + disabled state (existing `Button` `disabled:opacity-50` handles this); no full-composer skeleton.
- Send failure: `sonner` toast, `"Couldn't send your reply. Try again."` (public) / `"Couldn't save your note. Try again."` (internal) — composer content is preserved, not cleared, on failure.
- Auto-reopen (D-04): when a requester message triggers status `resolved`/`closed` → `open`, the `StatusChip` in `TicketMetaHeader` updates and a subtle inline system note appears in the thread: `"Ticket reopened — new reply from {contact}."` (`text-[12px] text-muted-foreground text-center py-1`, no avatar, no border — a lightweight system-event row distinct from both message variants above).

### 3. Contacts — `/contacts` (list) + `/contacts/[id]` (detail)

Not split-pane (D-08 calls out a "dedicated contact detail page" — full navigation, not a third column).

**`/contacts`** — single-column searchable list:
- Header row: page title "Contacts" (via `TopBar`, unchanged) + `Input` search (`Search` icon prefix, placeholder `"Search contacts…"`) below it, `px-6 py-4`.
- Rows (`ContactCard`, reuse `Card` `size="sm"`, stacked `space-y-2`, or a flat row list `divide-y divide-border` — flat rows preferred for scan-density): `Avatar` (initials, `bg-primary/10 text-primary`) + name (`text-[14px] font-medium`) + email (`text-[12px] text-muted-foreground`) + company (`text-[12px] text-muted-foreground`, `hidden md:block`) + ticket count badge (`Badge variant="secondary"`, e.g. `"3 tickets"`) + last-activity timestamp, right-aligned. Entire row is a `Link` to `/contacts/[id]`.

**`/contacts/[id]`** — dedicated detail page:
- Header card (`Card`, `p-6`): large `Avatar size="lg"` + name (`text-[18px] font-semibold`) + email/phone/company as a compact `dl`-style key-value list (`text-[13px]`) + free-form Notes field (`Textarea`, autosaves on blur via Server Action, label `"Notes"`).
- Below: `"Tickets from {name}"` (`text-[14px] font-medium text-muted-foreground`) heading, then a flat list of that contact's tickets reusing a **compact variant** of `TicketListRow` (1-line: subject + `StatusChip` + timestamp, no chip-row clutter, `Link` to `/tickets/[id]`).

**States:**
- List empty: `EmptyState` pattern, `Users` icon, heading `"No contacts yet"`, body `"Contacts are created automatically when someone submits a ticket. They'll show up here once your first request comes in."`
- Detail, contact with 0 tickets (rare — manually created): inline text `"No tickets from this contact yet."` (`text-[13px] text-muted-foreground py-4`), no full halo empty-state (too heavy for a sub-section).

### 4. Settings Surfaces (AIDA-12 partial) — extends `/settings`

Extend the existing `/settings` page (currently just "AI Features") with a horizontal `Tabs`-like sub-nav — **exception**: this is the one place `Tabs`-shaped navigation is justified (settings sections will keep growing in Phase 3/4), but implemented as simple `Link`s styled as pill tabs (same pattern as the inbox `FilterChipRow`, not a new dependency) rather than adding the shadcn `Tabs` package: `AI Features | SLA Policies | Tags | Custom Fields`, active = `bg-primary/10 text-primary`.

**SLA Policies** (D-14): a `Card` per priority (4 total, fixed — Low/Normal/High/Urgent, no add/delete), each row: priority label (`PriorityChip`) + two `Input type="number"` fields (First Response target, Resolution target) with a unit toggle/suffix ("hours"), `"Save SLA Targets"` `Button` per-row or one form-level `"Save SLA Targets"` for all four. Seeded defaults shown as placeholders/initial values: Urgent 1h/8h, High 4h/24h, Normal 8h/48h, Low 24h/72h (illustrative — RESEARCH.md, planner confirms final numbers).

**Tags** (D-17): flat list (`divide-y divide-border`), each row: `TagChip` + ticket-count (`text-[12px] text-muted-foreground`) + inline rename (`Input`, appears on click) + delete `Button variant="ghost" size="icon-sm"` (`Trash2` icon, `text-muted-foreground hover:text-destructive`, `aria-label="Delete tag {tag}"`). Delete triggers a `Dialog` confirmation (see Copywriting Contract — destructive actions).

**Custom Fields** (D-18): flat list, each row: label + `Badge variant="outline"` showing type (Text/Dropdown/Number/Checkbox/Date) + edit (`Button variant="ghost" size="icon-sm"`, `Pencil` icon, `aria-label="Edit field {label}"`) + delete (`Button variant="ghost" size="icon-sm"`, `Trash2` icon, `text-muted-foreground hover:text-destructive`, `aria-label="Delete field {label}"`, `Dialog` confirmation). "Add Field" `Button` opens a `Dialog` form: Label (`Input`), Type (`DropdownMenu` single-select of the 5 types), Options (only for Dropdown — repeatable `Input` list), primary action `"Create Field"`.

**States:**
- Tags empty: inline text `"No tags yet — tags created from the ticket composer will appear here for management."`
- Custom Fields empty: inline text `"No custom fields yet. Add one to start capturing extra ticket details."`

**Layout wrapper** (`PublicPageShell`, mirrors `(auth)/layout.tsx` exactly per DESIGN-SYSTEM §4.4's "layout wraps, page renders content only" rule — public pages are unauthenticated but must feel on-brand):

Card max-width **640px** (wider than the 400px auth Card — this is a content form, not a login box), `shadow-xl shadow-primary/5` per DESIGN-SYSTEM §4.5 "elevated card" pattern.

**Form fields** (D-19, `space-y-4` per DESIGN-SYSTEM §6): Name (`Input`), Email (`Input type="email"`), Subject (`Input`), Message (`Textarea`, `min-h-[120px]`), Attachments (drag-or-click zone, `border-2 border-dashed border-border rounded-lg p-4 text-center text-[13px] text-muted-foreground`, reuses the same 10MB/allowlist rules as the composer). No priority/category picker (D-19).

**Honeypot** (D-20, `HoneypotField`, no visual footprint): a text input named e.g. `company_website`, styled `absolute left-[-9999px] top-auto h-px w-px overflow-hidden`, `tabIndex={-1}`, `autoComplete="off"`, wrapped in a `<label>` with equally-hidden text (not `aria-hidden`, so real screen-reader users are still warned away, but visually and to naive bots it's just another field) — not `type="hidden"` per RESEARCH.md's guidance that visually-hidden traps more bots than a literal hidden input.

**Success confirmation** (same page, replaces the form on submit — not a redirect, to keep it simple and resilient): heading `"Request received"` (`text-[18px] font-semibold`), body `"We've sent a confirmation with a link to track your request."` (Claude's discretion: email delivery doesn't exist until Phase 3, so this copy is written to age well — see Assumptions), then a prominent `"View status"` `Button` linking directly to `/status/[token]` (shown immediately, not only via email, since Phase 2 has no outbound email yet).

**States:**
- Field validation errors: inline, via existing `Form`/`FormMessage` pattern (react-hook-form + zod), `text-[12px] text-destructive` under each field.
- Rate-limited (D-20): generic, non-alarming — `"You've submitted a few requests recently. Please wait a bit before trying again."` (`text-[13px] text-destructive`, shown as a form-level banner, not a field error).
- Honeypot tripped: **no visible error at all** — render the normal success state (D-20's "silent success" requirement — never tip off the bot).

### 6. Tokenized Public Status Page — `/status/[token]`

Same `PublicPageShell` wrapper, Card max-width **720px** (wider than the intake form — needs to fit a conversation thread).

- Header inside Card: `#{number}` + subject (`text-[18px] font-semibold`) + `StatusChip` (reuses the exact same component/tokens as the agent UI — consistency).
- Thread: same `ThreadMessage` component, **filtered to public-only** — internal notes are never fetched/rendered for this route (D-21, enforced server-side, not just hidden client-side — flag for planner: the query itself must exclude `visibility = INTERNAL`, never rely on UI filtering of a full payload).
- Follow-up composer: a reduced `Composer` — **no mode toggle** (there is no internal-note capability here), just a `Textarea` + `Send` `Button` (label `"Send follow-up"`) + attachment support (same 10MB/allowlist rules), protected by the same honeypot + rate-limit as the intake form (D-20).
- Submitting a follow-up on a `resolved`/`closed` ticket triggers auto-reopen (D-04) — same lightweight system-event row as §2 appears in the thread after refresh: `"Ticket reopened — new reply from {contact}."`

**States:**
- Invalid/expired token: full-page state (not inline) inside the Card — heading `"We couldn't find that ticket"`, body `"This status link may be invalid or expired. If you need help, please submit a new request."`, CTA `Button` `"Submit a new request"` → `/request`.
- Loading: `Skeleton` blocks inside the Card (header + 2 message shapes).

## Assumptions (review these)

Decisions made at the researcher's discretion where DESIGN-SYSTEM.md, CONTEXT.md, and RESEARCH.md did not fully specify — flagged for human review rather than blocking:

1. **Priority chip decluttering:** Low/Normal priority is hidden from ticket-list rows (shown only in the reading-pane header dropdown); only High/Urgent surface as list-row chips, to reduce per-row visual noise. Reconsider if agents report priority-blindness for Low/Normal tickets in UAT.
2. **Pagination, not virtualization:** long ticket lists use cursor-based `"Load more"` (default page size 50), not infinite scroll or list virtualization. Simpler for v1; revisit if inbox sizes in practice make this feel slow.
3. **Ticket detail routing:** `/tickets/[id]` opens a ticket within the same 2-pane layout (list stays visible, deep-linkable); `/tickets` alone shows the list with a "Select a ticket" reading-pane placeholder. Not explicitly specified in CONTEXT.md — this is the standard shared-inbox pattern (Front/Missive-style) referenced in CONTEXT's `<specifics>`.
4. **Public intake form path:** proposed as `/request` (not `/web-form` or `/contact`) for a clean customer-facing URL consistent with "request" terminology used elsewhere. Bikeshedding-safe to change.
5. **Date custom-field type:** rendered as a native `<input type="date">` (via the existing `Input` component), not a calendar/date-picker popover — avoids adding a new shadcn dependency for a single field type in v1.
6. **New `--warning`/`--success` oklch values** are proposed starting points tuned for text-legibility (mirroring how `--destructive` is used directly as `text-destructive`), not copied verbatim from DESIGN-SYSTEM.md §4.7's badge-background suggestion (which was tuned for a different usage — solid badge fill, not text-on-tint). Confirm visually in both light and dark mode before merging (DESIGN-SYSTEM §7 requirement). Both tokens deliberately follow the single-token `--destructive` convention (no `-foreground` companion) — every warning/success surface in this spec uses the tint pattern, never a solid fill, so no companion foreground token is ever needed.
7. **Success-page copy for the public form** ("We've sent a confirmation with a link to track your request.") pre-writes for Phase 3 email delivery even though no email exists yet in Phase 2 — the status link itself is shown immediately on the same page regardless, so the promise is not misleading (link is genuinely delivered, just not yet via email). Revisit copy if Phase 3 lands differently than expected.
8. **Contacts list row style:** flat divided rows (`divide-y`) rather than a card grid — optimizes for scan-density consistent with the ticket list, not explicitly mandated by CONTEXT.md.
9. **Settings sub-nav as pill-tab `Link`s** (not the shadcn `Tabs` package) — kept consistent with the inbox's `FilterChipRow` pattern and avoids adding a dependency used nowhere else in the app.
